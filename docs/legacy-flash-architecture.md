# ITMO Labs: Legacy Flash Architecture

## Назначение документа

Этот файл фиксирует базовую архитектуру исходной Flash/ActionScript версии проекта, чтобы упростить перенос логики в web-реализацию и помочь новым участникам быстро ориентироваться в старом коде.

## Точки входа

Главные Flash-документы:

- `sources/itmo-labs/Lab3.fla`
- `sources/itmo-labs/Lab4.fla`
- `sources/itmo-labs/Lab6.fla`

Связанные классы:

- `GUI.Lab3`, `GUI.Lab4`, `GUI.Lab6` - UI-классы, наследуются от `GUI.LabBase`;
- `LabApplication3`, `LabApplication4`, `LabApplication6` - контроллеры лабораторий, наследуются от `LabApplicationBase`.

## State Machine

`LabApplicationBase` выступает как центральная state machine лаборатории.

Основные состояния:

- `Start`
- `Dragging`
- `Piping`
- `Elbows`
- `Eraser`
- `Instruments`
- `TurningOn`
- `Running`

Выбор пользовательского поведения зависит от пары `(state, equipmentID)` через `getBehaviour()`.

## Основные уровни модели

### GUI слой

`GUI.LabBase` содержит:

- tiled plane и области размещения;
- группы кнопок для фаз схемы и выполнения;
- экземпляр `SchemeBase`-совместимого класса;
- панели миниатюр оборудования;
- графические сенсоры и оверлеи.

Для конкретных лабораторий используются соответствующие GUI-классы и схемы.

### Scheme слой

`SchemeBase` - основная модель схемы. Отвечает за:

- хранение оборудования;
- проверку корректности схемы;
- линковку соединений;
- проверку условий запуска;
- сценарии перехода к выполнению.

Связанные классы:

- `Scheme3`
- `Scheme4`
- `Scheme6`
- `Scheme3And4Base`

### Equipment слой

`EquipmentBase` представляет физический объект на схеме.

Он связывает:

- графический clip;
- сокеты/точки подключения;
- размещение на tiled-плоскости;
- установку приборов и специфическое состояние оборудования.

## Graphics hierarchy

Графическая иерархия строится вокруг `Graphics.Common.Equipment` и производных клипов, например:

- `Graphics.Boiler`
- `Graphics.Tank`
- `Graphics.Compressor`
- `Graphics.Condenser`
- `Graphics.Heater`
- и другие элементы установки.

## Socket network

Сетевая модель потока строится на классе `Socket`.

Сокеты хранят физические величины, включая:

- давление `P`;
- массовый расход `G`;
- энтальпию `h`;
- температуру `T`.

Когда сокеты совмещаются на сетке, между ними устанавливается связь. Изменение параметров может распространяться по связанной сети.

## ID и фабрика объектов

Класс `ID.as` централизует идентификаторы оборудования и лабораторий.

Примеры:

- ID оборудования: Boiler, Chamber, Compressor, Condenser, Elbow, Flowmeter, Glass, Heater, Pipe, Receiver, Tank, Valve, Nozzle;
- ID лабораторий: Lab3, Lab4, Lab6.

`ID.createObjectById()` используется как фабрика для создания оборудования, включая lab-specific варианты.

## Behaviour pattern

Взаимодействие пользователя разнесено по behaviour-классам в `sources/itmo-labs/Behaviours/`.

Примеры:

- `EquipmentDragging`
- `LabsPiping`
- `ValveRunning`
- `EquipmentTurningOn`
- `EquipmentEraser`

Это важная часть переноса: в web-версии аналогичное поведение, скорее всего, должно быть выражено как явные режимы взаимодействия и операции над доменной моделью.

## Важные правила валидации

Среди ключевых правил, замеченных в legacy-коде:

- корректное расположение glass относительно condenser;
- специальные проверки порядка запуска;
- защита от некорректного включения heater раньше boiler.

При переносе поведения в web-реализацию эти правила надо явно отражать в валидаторах и сценариях запуска.
